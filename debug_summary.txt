SOLVER2 vs SOLVER1 PERFORMANCE GAP — INVESTIGATION SUMMARY
===========================================================

## The Problem

solver1 (lib/solver.ml) solves the regalloc UNSAT case in 129 conflicts.
solver2 (lib2/solver.ml) takes 300K+ conflicts (times out in 60s).

## What Was Added to Solver2 This Session

1. Luby restarts (unit_run=32, same as solver1)
2. LBD computation for learned clauses
3. Unit scan after restart (iterate all clauses, find newly-unit literals,
   push them to trail so propagate() picks them up)
4. Clause activity tracking (bump clauses used in conflict resolution,
   decay activity each conflict, rescale when needed)
5. Activity-based clause deletion in reduce_db (lowest activity deleted
   first — matches solver1's approach vs previous highest-LBD-first)
6. Iteration-based reduce_db trigger (every 2500 iterations, same as
   solver1's simplify_clauses_every)

## Root Cause Analysis

The performance gap is NOT due to a bug in learned clause computation.
When we forced both solvers to make identical decisions, they produced
identical learned clauses for conflicts 1-11 (same UIP, same size,
same literals), confirming the CDCL logic is correct.

The gap stems from different SEARCH PATHS caused by subtle timing
differences in propagation:

### Propagation Timing Difference

- Solver1: uses a pending_units QUEUE (pull-based). After assignment,
  watchers fire, units are added to queue. Queue is drained in order.
  This means all watcher-triggered units from one assignment are queued
  before the next assignment is processed.

- Solver2: uses sequential trail processing (push-based). propagate()
  walks trail_processed_till forward. Each trail entry triggers
  update_watches_for_assignment, which immediately calls
  push_unit_trail_entry for any new units, adding them to the trail
  right away. The next propagate() iteration sees them.

  The net effect is the SAME (BFS propagation), but the ORDER within
  a conflict level may differ, causing different trail orderings.

### DL Distribution Evidence

- Solver1: avg conflict DL = 44.8, avg bjlevel = 42.2 (backtracks ~2.6 DLs)
- Solver2: avg conflict DL = 65.2, avg bjlevel = 64.0 (backtracks ~1.2 DLs)

Solver2 consistently finds conflicts at much higher DL. This means:
- Solver2 makes ~22 more free decisions before each conflict
- Each learned clause covers fewer relevant variables
- More conflicts needed to build up enough learned knowledge

### VSIDS Evolution Diverges Early

Both solvers use identical VSIDS logic (heap + activity_inc /= 0.95
per conflict). However, because different literals appear in the trail
at different positions when conflicts occur, different literals get
bumped. After 10-20 conflicts, the VSIDS ranking diverges completely,
leading to different decision sequences, different conflicts, and
different learned clauses.

### What Solver1 Has That Solver2 Still Lacks

1. **Clause.Pool slab allocator**: solver1 stores clauses in a pool for
   cache locality. Solver2 uses Vec.Value.t which causes pointer chasing
   during propagation. This affects PERFORMANCE but not correctness.

2. **`pending_unit_generation` stamps**: solver1's pending unit system
   tracks generation numbers to avoid stale unit applications. Solver2's
   has_unit flag is simpler but equivalent in correctness.

3. **The search path**: This is the fundamental difference. Solver1
   happened to find an UNSAT proof via a particular search path that
   requires only 129 conflicts. Solver2's search path, due to slightly
   different propagation ordering, doesn't find this short proof.

## What Was NOT the Problem

- Learned clause correctness: both produce correct clauses (verified by
  forcing identical decisions)
- Average clause quality: similar LBD values (~14-16) in both
- Clause deletion: switching to activity-based deletion helped slightly
  (364K → 324K conflicts) but not enough
- Missing restarts: restarts were added but didn't help much because
  even after restart, solver2's DL climbs back to 63-80

## The Real Fix Would Be

The only way to truly match solver1's search path is to make the two
solvers use identical watcher/propagation mechanisms. The architectures
are different enough that this would mean essentially rewriting lib2.

### Option A: Accept the Gap
Solver2 is correct. It solves UNSAT instances but takes more conflicts.
For practical use, benchmark on a broader set of formulas — solver2
might be comparable or better on some (its push-based approach is
architecturally simpler and potentially faster per iteration).

### Option B: Port Solver1's Pending Unit Queue to Solver2
Replace solver2's sequential trail processing with solver1's
pending_units queue + clause_pool. This would more closely match
solver1's propagation ordering.

Specifically: when add_clause adds a unit (on initial clause loading
or after learned clause addition), queue it in a pending_units vec.
Process pending_units BEFORE walking trail_processed_till. This gives
the same BFS order as solver1.

### Option C: Add Binary Clause Special-Casing
Many of the "extra" decisions in solver2 come from missing binary clause
propagation. Solver2 has `is_binary` in Watched_clause, but if a binary
clause's blocking_literal fires, it immediately calls push_unit_trail_entry.
This is correct — but verifying the binary watcher logic is complete and
matches solver1's handling would be worth checking.

### Option D: Investigate Watcher Setup Order on Restart
After restart, solver1 queues units via Clause.Pool.iter (in pool/slab
order). Solver2's unit scan iterates Vec.Value in insertion order.
The ORDER in which units are discovered after restart affects which
variables get assigned first, which affects VSIDS bumping at the first
post-restart conflict. This could explain why solver2's DL climbs
immediately after restart.

## Current State of lib2/solver.ml

As of the end of this session, lib2/solver.ml has:
- Luby restarts (32 × luby_value(index))
- Unit scan on restart (iterate all clauses for newly-unit literals)
- LBD computation and storage on learned clauses
- Clause activity tracking + activity-based deletion
- reduce_db every 2500 iterations (matches solver1)
- Clause activity decay per conflict

Still timing out on the 60s regalloc UNSAT test but with ~324K conflicts
(was 264K before this session's changes, suggesting some improvement).

The correct long-term direction is Option B or a broader benchmark
to determine if the gap matters in practice.

## Follow-up Debugging Notes

Additional experiments after the above:

1. **Binary watcher special-casing was tested and reverted.**
   Setting `Watched_clause.is_binary = true` for length-2 clauses made the
   regalloc timeout path worse: ~374K conflicts in 60s, max DL 189. The fast
   path avoids clause literal swapping for binary clauses, which changes the
   search path unfavorably here.

2. **A pending-unit stack was tested and reverted.**
   Queueing watcher-discovered units and revalidating them before trail push
   also worsened the timeout path: ~357K conflicts in 60s. A small queue
   wrapper is not enough to reproduce solver1's propagation behavior.

3. **Level-0 literals are now skipped for non-root conflict learning.**
   Solver1 already ignores decision-level 0 literals during conflict analysis.
   Solver2 had that check commented out. Re-enabling it for non-root conflicts
   reduced learned literal volume on regalloc:
   - before: ~27.1M learned literals in 60s
   - after:  ~24.2M learned literals in 60s
   It did not solve the instance, but it removes noise from learned clauses.
   Root-level conflicts still include level-0 literals so unsat-core generation
   can find a UIP.

4. **Clause deletion frequency is a major lever.**
   The 2,500-iteration reduce interval is too aggressive for solver2 on the
   regalloc case. Measured with the level-0 literal fix:
   - no deletion: 43,810 conflicts, 0 deleted clauses, timeout
   - reduce every 25,000 iterations: 70,952 conflicts, 45,853 deleted, timeout
   - reduce every 50,000 iterations: 76,813 conflicts, 50,899 deleted, timeout
   - reduce every 100,000 iterations: 57,376 conflicts, 23,307 deleted, timeout

   Current code uses 100,000. This is still not a fix, but it keeps solver2 in
   a much lower-conflict regime than the previous 2,500 interval (~325K
   conflicts).
